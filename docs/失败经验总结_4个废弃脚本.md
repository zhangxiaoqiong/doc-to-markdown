## 二、详细失败案例分析

### 2.1 失败 1：convert_docs_multimodal.py

#### 2.1.1 设计思路
```
期望的处理流程：
PDF → pdf2image → 转为图片列表 → Vision API → Markdown
```

#### 2.1.2 核心代码
```python
try:
    from pdf2image import convert_from_path
except ImportError:
    print("需要安装 pdf2image: pip install pdf2image")
    sys.exit(1)

def pdf_to_images(file_path):
    """将PDF转换为图片列表"""
    try:
        images = convert_from_path(file_path, dpi=150)  # ← 这行会失败
        return images
    except Exception as e:
        print(f"[警告] 转换PDF为图片失败: {str(e)}")
        return []
```

#### 2.1.3 失败过程

**运行日志**：
```
$ python convert_docs_multimodal.py

[警告] 转换PDF为图片失败: Unable to get page count. Is poppler installed and in PATH?
[警告] 无法将PDF转换为图片: 丰图科技员工费用报销操作指引V3.0 (1).pdf
[警告] 转换PDF为图片失败: Unable to get page count. Is poppler installed and in PATH?
[警告] 无法将PDF转换为图片: 丰图科技客户评级管理规则【1.0】.pdf
...
```

#### 2.1.4 根本原因

```
pdf2image 库需要 poppler 工具
    ↓
poppler 是 C/C++ 依赖，不是 Python 包
    ↓
Windows 上需要手动下载并配置 PATH
    ↓
用户环境变量配置复杂，容易出问题
    ↓
即使安装成功，也可能版本不兼容
```

**实际错误**：
```
ImportError: Unable to import pdf2image.
This is likely due to poppler not being installed or not being on your PATH.
On Mac: brew install poppler
On Linux: apt-get install poppler-utils
On Windows: (需要手动下载 poppler 并添加到 PATH)
```

#### 2.1.5 为什么放弃

| 问题 | 影响 | 严重性 |
|------|------|--------|
| **环境依赖复杂** | 需要安装外部工具 | ⭐⭐⭐ 高 |
| **跨平台兼容性差** | Windows/Mac/Linux 配置不同 | ⭐⭐⭐ 高 |
| **步骤太多** | PDF → 图片 → Vision API → 合并 | ⭐⭐ 中 |
| **内存占用高** | 大量图片加载到内存 | ⭐⭐ 中 |
| **有更优方案** | Claude Vision 直接支持 PDF | ⭐⭐⭐⭐⭐ 决定性 |

#### 2.1.6 教训

✅ **优先选择零依赖或最少依赖的方案**
✅ **避免跨平台的 C/C++ 依赖**
✅ **充分利用 API 的原生能力**（Vision 直接支持 PDF！）

### 2.2 失败 2：convert_pdf_vision.py

#### 2.2.1 设计思路
```
期望的处理流程：
PDF → PyMuPDF (fitz) → 转为图片 PNG → Vision API → Markdown
```

#### 2.2.2 核心代码
```python
import fitz  # PyMuPDF

def pdf_to_images(file_path):
    """用 PyMuPDF 将 PDF 转换为图片"""
    images = []
    try:
        pdf_doc = fitz.open(file_path)  # ← 这个会成功
        for page_num, page in enumerate(pdf_doc):
            # 渲染为图片
            pix = page.get_pixmap(matrix=fitz.Matrix(1.5, 1.5))
            image_data = pix.tobytes("png")
            images.append(image_data)
        pdf_doc.close()
    except Exception as e:
        print(f"[错误] PDF转换失败: {str(e)}")
        return []
    return images
```

#### 2.2.4 根本原因

```
pip install pymupdf 下载的版本有问题

内部模块缺失（frontend）

或者与 miniforge 的 Python 版本不兼容
```

#### 2.2.5 为什么放弃

|------|------|--------|
| **无法导入** | 脚本完全无法运行 | ⭐⭐⭐⭐⭐ 极高 |
| **版本兼容问题** | miniforge + PyMuPDF 冲突 | ⭐⭐⭐ 高 |
| **调试困难** | 需要重新卸载/安装/对比版本 | ⭐⭐⭐ 高 |
| **时间成本** | 环境问题排查很耗时 | ⭐⭐⭐ 高 |

#### 2.2.6 教训

✅ **避免使用有导入问题的库**
✅ **优先用经过验证的稳定库**
✅ **不要为了"完整控制"而选择复杂库**（PyMuPDF 比 pdfplumber 复杂）
✅ **如果库本身就有问题，不值得继续尝试**

### 2.3 失败 3：convert_split_pdf.py（第 1 版分割方案）

#### 2.3.1 设计思路
```
期望的处理流程：
20MB PDF → 分割成 50 页一块 → 逐块 Vision API → 合并
```

#### 2.3.2 核心代码
```python
def split_pdf(file_path, pages_per_chunk=50):  # ← 块太大！
    """将PDF分割成多个部分"""
    reader = PdfReader(file_path)
    total_pages = len(reader.pages)

    chunks = []
    for start_page in range(0, total_pages, pages_per_chunk):
        end_page = min(start_page + pages_per_chunk, total_pages)

        # 这会创建很大的块（50 页 = 14-16MB）
        writer = PdfWriter()
        for page_num in range(start_page, end_page):
            writer.add_page(reader.pages[page_num])

        chunks.append(chunk_path)

def process_large_pdf():
    chunks = split_pdf(file_path, pages_per_chunk=50)  # ← 问题在这

    for chunk in chunks:
        markdown = convert_pdf_chunk(chunk)  # ← API 调用仍然超限
```

#### 2.3.3 失败过程

**运行日志**：
```
$ python convert_split_pdf.py

开始处理: 员工报销管理规定.pdf

第1步: 分割PDF...
总页数: 10
  已分割: 第 1-10 页

已分割为 1 个部分

第2步: 逐块转换...
[1/1] 转换第 1-10 页...
  [警告] 转换失败: <!DOCTYPE html>
<html>
...
<h1>An error occurred.</h1>
<p>Sorry, the page you are looking for is currently unavailable.</p>
...
```

#### 2.3.4 根本原因

```
pages_per_chunk=50 太大了
    ↓
50 页的 PDF 编码后 base64 大小 ~14-16MB
    ↓
超过了 Claude API 的文件大小限制（~10-12MB）
    ↓
API 返回 503 错误（服务暂时不可用）
```

**计算**：
```
10 页 PDF：
  ├─ 原始大小：~2MB
  ├─ Base64 编码：2MB × 1.33 = 2.66MB （可以）

10 页当成 1 个块：不问题

但原文件是 20MB：
  ├─ 分成 50 页一块（全部 10 页）
  ├─ Base64 编码：20MB × 1.33 = 26.6MB
  └─ 超过 API 限制！

结果：0 个块成功，0/1 转换率
```

#### 2.3.5 为什么放弃

| 问题 | 影响 | 严重性 |
|------|------|--------|
| **分块太大** | 仍然超过 API 限制 | ⭐⭐⭐⭐⭐ 极高 |
| **全部失败** | 没有一个块成功 | ⭐⭐⭐⭐⭐ 极高 |
| **需要调整** | 必须改变分块大小 | ⭐⭐⭐ 中 |
| **重试后成功** | 改成 5 页一块就成了 | ✅ 有解决方案 |

#### 2.3.6 解决方案

```python
# 错误的做法
pages_per_chunk = 50   # ❌ 块 = 14-16MB，超限

# 正确的做法
pages_per_chunk = 5    # ✅ 块 = 1.4-1.6MB，安全
```

**改进后的结果**：
```
$ python convert_split_pdf_v2.py

处理: 员工报销管理规定.pdf

第1步: 分割PDF...
已分割为 2 个部分

第2步: 逐块转换...
[1/2] 第 1-5 页... ✓
[2/2] 第 6-10 页... ✓

已成功转换: 2/2 个部分
```

#### 2.3.7 教训

✅ **必须精确计算分块大小**
✅ **Base64 编码会增加 33% 体积**
✅ **保守估计，预留安全余量**
✅ **小块多次 > 大块少次**（网络稳定性）
✅ **测试很重要，一个参数改错全部失败**

### 2.4 失败 4：convert_large_pdf.py

#### 2.4.1 设计思路
```
期望的处理流程：
20MB PDF → 直接读成 base64 → Vision API 一次调用
```

```python
def pdf_to_base64(file_path):
    """将PDF文件转换为base64"""
    with open(file_path, 'rb') as f:
        return base64.b64encode(f.read()).decode("utf-8")

def convert_large_pdf(file_name, file_path):
    """处理超大PDF"""
    client = Anthropic(...)

    pdf_base64 = pdf_to_base64(file_path)

    print(f"正在处理超大PDF: {file_name}（大小: {file_size_mb:.1f}MB）...")

    # ← 直接发送，没有分割
    message = client.messages.create(
        model="claude-opus-4-6",
        messages=[{
            "content": [
                {...},
                {
                    "type": "document",
                    "source": {
                        "type": "base64",
                        "media_type": "application/pdf",
                        "data": pdf_base64  # ← 26.6MB 直接发送
                    }
                }
            ]
        }]

```

#### 2.4.3 失败过程

**运行日志**：
```
$ python convert_large_pdf.py

正在处理超大PDF: 员工报销管理规定.pdf（大小: 26.1MB）...

[失败] <!DOCTYPE html>
```

#### 2.4.4 根本原因

```
20MB PDF 文件
    ↓
Base64 编码 → 26.6MB
    ↓
发送给 Vision API
    ↓
超过单次请求文件大小限制 (~10-12MB)
    ↓
API 服务返回 503 错误（服务暂时不可用）
```

#### 2.4.5 为什么失败

| 问题 | 影响 | 严重性 |
|------|------|--------|
| **文件太大** | 直接发送无法处理 | ⭐⭐⭐⭐⭐ 极高 |
| **没有备选方案** | 不分块就无路可走 | ⭐⭐⭐⭐⭐ 极高 |
| **无法重试** | 文件太大，重试也是同样错误 | ⭐⭐⭐⭐ 高 |
| **必须分块** | 唯一解决方案 | ⭐⭐⭐⭐⭐ 决定性 |

#### 2.4.6 教训

✅ **单次 API 请求有文件大小限制**
✅ **对于超大文件，分块处理是必须的**
✅ **不要假设"试试看"能成功**
✅ **提前计算，做好规划**

## 六、最重要的教训

### 6.1 技术选择的优先级

```
优先级 1：选择最简单的方案
  ├─ 能一个 API 调用搞定，就不要多个工具配合
  └─ Vision API 直接支持 PDF ✅

优先级 2：选择零依赖或最少依赖
  ├─ 避免 C/C++ 依赖（poppler）❌
  ├─ 避免有 bug 的库（PyMuPDF）❌
  └─ 优先 Python 原生库 ✅

优先级 3：充分理解 API 能力
  ├─ 不要假设"应该支持"
  ├─ 要实际查阅文档
  └─ Vision API 支持 PDF document 类型是关键发现！✅

优先级 4：精确计算参数
  ├─ pages_per_chunk = 50 ❌（超限）
  ├─ pages_per_chunk = 5 ✅（成功）
  └─ 1 个参数的差异可能决定成败
```

### 6.2 避免的陷阱

| 陷阱 | 表现 | 后果 |
|------|------|------|
| **过度工程化** | 用复杂工具解决简单问题 | 增加依赖和风险 |
| **猜测 API 能力** | 假设"应该支持 PDF" | 浪费时间开发中间层 |
| **参数不经计算** | 直接用默认值或随意设置 | 失败调试困难 |
| **不充分测试** | 大块处理不测试直接用 | 全部请求都失败 |
| **放弃太快** | 一个方案不行就换 | 浪费时间，学不到东西 |

### 6.3 成功的必要条件

✅ **充分的技术探索**（尝试 4 个失败方案）
✅ **逐步改进**（分块大小从 50 → 5）
✅ **精确的计算**（明确知道 5 页 = 1.4MB）
✅ **完整的设计**（分割 + 处理 + 合并）
✅ **充分的测试**（能看到实时日志和错误）
